## Repository layout

Standalone extensions live at the root. Packs and collections may contain sub-extension folders.

```text
sindri-extensions/
├── index.json                    ← registry index (auto-generated — do not edit)
├── manifest.schema.json          ← JSON Schema; IDEs validate manifest.json against this
├── CONTRIBUTING.md               ← this file
│
├── sindri-commit-streak/         ← standalone code extension
│   ├── manifest.json
│   ├── README.md
│   ├── package.json
│   ├── tsconfig.json
│   ├── src/
│   │   ├── extension.ts          ← host-side entry point (exports activate/deactivate)
│   │   └── webview/              ← webview source (only if using registerWebviewPanel)
│   │       ├── index.ts(x)
│   │       └── components/
│   └── dist/                     ← compiled output (gitignored)
│
├── aurora-theme-pack/            ← extension pack
│   ├── manifest.json             ← extensionPack lists the three sub-extensions
│   ├── aurora-theme/             ← sub-extension (leaf)
│   ├── aurora-file-icons/        ← sub-extension (leaf)
│   └── aurora-ui-icons/          ← sub-extension (leaf)
│
├── community-theme-collection/   ← collection (pack of packs)
│   ├── manifest.json             ← extensionPack lists 15 theme packs
│   ├── sindri-community-icons-base/  ← standalone base extension (Icon Theme Base)
│   │   ├── manifest.json
│   │   ├── icons.json
│   │   └── icons/                ← 46 SVG templates (CSS custom properties for palette)
│   ├── sindri-dracula/           ← one of 15 theme packs
│   │   ├── manifest.json         ← extensionPack: [dracula-color, dracula-icons]
│   │   ├── sindri-dracula-color/ ← leaf: color theme
│   │   └── sindri-dracula-icons/ ← leaf: icon theme (extends community-icons-base)
│   └── …
│
└── …
```

`index.json` is regenerated by `bun run scripts/build-index.ts`. **Do not edit it by hand.**

## Extension types at a glance

| Category | Needs extension host? | What it provides |
| --- | --- | --- |
| **Color Theme** | No | CSS custom-property palette applied from a ThemeDef JSON file |
| **File Icon Theme** | No | File-tree icons — an IconThemeDef JSON with SVGs mapped to extensions and file names |
| **UI Icon Theme** | No | Activity bar and toolbar glyphs — a UiIconPackDef JSON mapping panel IDs to SVG strings |
| **Extension Pack** | No | A curated bundle that installs a list of extensions together |
| **Language Support** | Yes | Tree-sitter grammar, LSP language server, DAP debugger adapter |
| **Language Pack** | Yes | Human-language UI localisation |
| **Test & Task Adapter** | Yes | Test runner or build task adapter |
| **UI Extension** | Yes | Custom panels, status bar items, editor decorations, commands |

Data-only extensions (Color Theme, File Icon Theme, UI Icon Theme, Extension Pack) install today. Extensions that require the host are wired up once the QuickJS host ships — you can scaffold them now and they will activate automatically.

## Adding an extension

1. Create a new folder at the repository root with a short, lowercase, hyphen-separated name (e.g. `my-rust-theme`).
2. Add `manifest.json` with `"$schema": "../manifest.schema.json"` — this gives IDE validation and autocompletion.
3. Add `README.md` — this is the detail page shown when a user clicks your extension in the marketplace.
4. For data extensions: add the asset files referenced from `manifest.json`.
5. For code extensions: follow the [Code extension guide](#code-extension-guide) below.
6. Open a pull request. CI validates the manifest and regenerates `index.json`.

## Release pipeline

This repo uses **[Changesets](https://github.com/changesets/changesets)** for version management. Each extension is versioned independently — shipping a theme pack doesn't bump any code extension. The CI pipeline is split across three workflows:

| Workflow | Trigger | What it does |
| --- | --- | --- |
| `pr-check.yml` | Every PR to `main` | Enforces changeset presence, builds changed extensions, validates `index.json` |
| `release.yml` | Push to `main` | Creates a "Version PR" (or publishes, if the Version PR was just merged) |
| `nightly.yml` | 02:00 UTC daily | Builds every extension and uploads `{id}-nightly.sinxt` to the rolling `nightly` release |

### Branching model

```text
feature/my-change  ──┐
hotfix/fix-crash   ──┤─── PR → main  (always squash-merged)
chore/update-deps  ──┘
```

There is **one long-lived branch: `main`**. All work — features, bugfixes, hotfixes, chores — lands on `main` via PR. There is no `develop` branch.

- **Stable releases**: Changesets Version PR merged to `main` → `release.yml` publishes.
- **Beta releases**: Set version to `x.y.z-beta.N` in the changeset (see [Beta releases](#beta-releases)).
- **Nightly**: Rolling build off `main` HEAD every night — no version bump, no changeset needed.

### Adding a changeset

Every PR that modifies an extension **must** include a changeset file. The `pr-check` CI job enforces this; add `[skip changeset]` to your PR title only for docs/tooling changes that don't affect extension behaviour.

```sh
# From the repository root
bun run changeset
```

This is an interactive prompt. Select the extensions you changed and choose a bump type:

| Bump | When to use |
| --- | --- |
| `patch` | Bug fixes, asset updates, small tweaks |
| `minor` | New features, new contributes entries |
| `major` | Breaking changes to the public API surface |

Commit the generated `.changeset/*.md` file alongside your code changes.

### Version PR flow

When changesets accumulate on `main`, the Changesets action automatically opens (or updates) a PR titled **"chore: release packages"**. This PR shows:

- Which extensions will be bumped
- What their new versions will be
- A generated `CHANGELOG.md` update

**Review it, then merge it.** On merge, `release.yml` runs `bun run release` which:

1. Reads each extension's bumped `manifest.json` version
2. Checks whether a GitHub Release tagged `{id}-v{version}` already exists
3. Builds the `.sinxt` bundle for any that don't
4. Creates a GitHub Release with the `.sinxt` as the asset

> ⚠️ **Do not merge the Version PR until all PRs for the intended release are in.** The Version PR is the release gate — merging it triggers publishing.

### Beta releases

To ship a pre-release for a specific extension without affecting others:

1. Run `bun run changeset` and select only the extension under development.
2. Edit the generated `.changeset/*.md` file and set the version field to a pre-release identifier: `1.2.0-beta.1`.
3. Commit and open a PR normally. The Version PR will show this as a pre-release bump.
4. When the Version PR merges, `create-releases.ts` detects the `-beta.N` suffix and creates the GitHub Release with the `--prerelease` flag.

Sindri's update checker surfaces pre-releases only to users who have opted into beta updates for that extension (or have it installed already at a pre-release version).

### Nightly builds

The `nightly.yml` workflow runs at 02:00 UTC every day. It:

1. Builds every extension with `--bundle`
2. Renames each `{id}-{version}.sinxt` → `{id}-nightly.sinxt`
3. Uploads all assets to the rolling `nightly` GitHub Release (clobbering the previous day's build)
4. Force-updates the `nightly` git tag to `HEAD`

Nightly assets have a **stable URL** — the `nightly` release tag never changes, only its assets do. Sindri uses this to offer nightly channel subscriptions per extension.

No changeset is required for nightly — it builds whatever is on `main` at 02:00 UTC.

## manifest.json field reference

### `id`

**Required.** `publisher.name` format — lowercase letters, numbers, and hyphens only. Globally unique.

```json
"id": "yourname.my-extension"
```

---

### `name`, `version`, `publisher`, `description`

**All required.**

| Field | Type | Notes |
| --- | --- | --- |
| `name` | string | Display name in the marketplace card |
| `version` | string | `major.minor.patch` — should match a git tag |
| `publisher` | string | Matches the prefix of `id` |
| `description` | string | One sentence shown in the marketplace card |

---

### `categories`

**Required.** At least one value from:

```text
"Color Theme" | "File Icon Theme" | "UI Icon Theme" | "Extension Pack"
"Language Support" | "Language Pack" | "Test & Task Adapter" | "UI Extension"
```

---

### `permissions`

**Required.** List of `sindri.*` namespaces the extension may call. Data-only extensions pass `[]`.

| Permission | What it grants |
| --- | --- |
| `"sindri.env"` | Shell execution (`exec`), filesystem access (`fs`), `workspaceRoot` |
| `"sindri.ui"` | Status bar items, tree views, webview panels, quick picks |
| `"sindri.lsp"` | Register Language Server Protocol adapters *(planned)* |
| `"sindri.dap"` | Register Debug Adapter Protocol adapters *(planned)* |

Sindri shows each requested permission to the user before installation. Request only what you need.

```json
"permissions": ["sindri.env", "sindri.ui"]
```

---

### `engines`

**Required.** Minimum Sindri version.

```json
"engines": { "sindri": ">=0.1.0" }
```

---

### `contributes`

**Required** (all sub-fields optional).

| Sub-field | Type | Description |
| --- | --- | --- |
| `contributes.themes[]` | Color Theme | ThemeDef JSON file references |
| `contributes.iconThemes[]` | File Icon Theme | IconThemeDef JSON file references |
| `contributes.uiIconPacks[]` | UI Icon Theme | UiIconPackDef JSON file references |
| `contributes.treeViews[]` | UI Extension | Declare a Tier 1 tree-view panel (backed by `registerTreeView`) |
| `contributes.webviewPanels[]` | UI Extension | Declare a Tier 2 webview panel (backed by `registerWebviewPanel`) |
| `contributes.lsp[]` | Language Support | LSP adapter declarations |
| `contributes.dap[]` | Language Support | DAP adapter declarations |
| `contributes.grammars[]` | Language Support | Tree-sitter grammar file references |
| `contributes.taskAdapters[]` | Test & Task Adapter | Task runner adapter declarations |
| `contributes.configuration` | Any | Settings schema contributed to the Settings panel |

---

### `main`

Path to the compiled extension entry-point bundle. **Omit for data-only extensions.**

```json
"main": "dist/extension.js"
```

---

### `extensionPack`, `tags`

Optional. `extensionPack` lists extension IDs bundled by an Extension Pack. `tags` are freeform search facets.

```json
"extensionPack": ["sindri.aurora-theme", "sindri.aurora-file-icons"],
"tags": ["dark", "aurora", "violet"]
```

## Code extension guide

A code extension is a **UI Extension** (or Language Support, Language Pack, etc.) with a `main` entry point. The extension host runs the compiled JavaScript and injects the `sindri` global.

### Project layout

```text
my-extension/
├── manifest.json
├── README.md
├── package.json          ← devDependency: @sindri/api
├── tsconfig.json
├── src/
│   ├── extension.ts      ← must export activate() and deactivate()
│   └── webview/          ← only needed if you use registerWebviewPanel
│       ├── index.ts(x)   ← webview entry point
│       ├── components/   ← UI components
│       └── lib/          ← pure utilities and types
└── tests/                ← optional unit tests (bun test)
```

**`package.json`** — install the API types as a dev dependency:

```json
{
  "name": "my-extension",
  "private": true,
  "type": "module",
  "devDependencies": {
    "@sindri/api": "file:../../../sindri-ide/packages/sindri-api"
  }
}
```

**`tsconfig.json`** — include the ambient types explicitly:

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "isolatedModules": true,
    "noEmit": true
  },
  "include": ["src", "node_modules/@sindri/api/index.d.ts"]
}
```

> `"types"` / `"typeRoots"` do not work for `@sindri/api` because it is not under `@types/`. Use `include` instead — same pattern as `@types/vscode`.

---

### Build pipeline

All code extensions are built with the `build-extension.ts` script from the `sindri-ide` directory:

```sh
cd sindri-ide
bun run scripts/build-extension.ts ../sindri-extensions/my-extension
```

**What it produces:**

| Output | When | Notes |
| --- | --- | --- |
| `dist/extension.js` | Always | Host-side bundle — CJS, platform: node |
| `dist/webview.js` | When `src/webview/index.*` exists | IIFE, platform: browser |
| `dist/webview.css` | When the webview imports `.scss` | Extracted by esbuild-sass-plugin |

**Framework detection** — the build script reads your `package.json` dependencies and configures JSX automatically:

| Dependency | JSX config applied |
| --- | --- |
| `solid-js` | `jsxImportSource: "solid-js"` |
| `react` | `jsx: "automatic"`, `jsxImportSource: "react"` |
| `preact` | `jsx: "automatic"`, `jsxImportSource: "preact"` |
| None | No JSX (plain TypeScript) |

**Svelte** — add `svelte` as a dependency; the script detects it and adds the `esbuild-svelte` + `svelte-preprocess` plugins. Svelte's `<style>` blocks are compiled inline; SCSS imports go through esbuild-sass-plugin and produce `dist/webview.css`.

### Extension API reference

The `sindri` global is injected by the host — no import needed.

#### `sindri.commands`

```typescript
// Register a command the user can invoke from the command palette
sindri.commands.register("my-ext.do-thing", () => { /* ... */ });

// Execute a command programmatically
await sindri.commands.execute("my-ext.do-thing");
```

#### `sindri.env`

```typescript
// Absolute path to the open workspace folder, or null
const root: string | null = sindri.env.workspaceRoot;

// Run a subprocess — returns { code, stdout, stderr }
const { stdout, code } = await sindri.env.exec("git", "log", "--oneline", "-5");

// Filesystem (scoped to workspace)
const content = await sindri.env.fs.read("src/main.ts");
const files   = await sindri.env.fs.glob("src/**/*.ts");
const exists  = await sindri.env.fs.exists("Cargo.toml");
```

#### `sindri.ui`

```typescript
// Status bar item
const item = sindri.ui.createStatusBarItem("my-ext.chip", { text: "…", tooltip: "…" });
item.text = "Hello";
item.show();

// Tree view (Tier 1 — declare in contributes.treeViews)
sindri.ui.registerTreeView("my-ext.my-tree", { treeDataProvider: myProvider });

// Quick pick (blocking)
const picked = await sindri.ui.showQuickPick(
  [{ label: "Option A" }, { label: "Option B" }],
  { title: "Choose one", placeholder: "Type to filter…" },
);

// Quick pick (event-driven)
const qp = sindri.ui.createQuickPick();
qp.items = [{ label: "Item 1" }, { label: "Item 2" }];
qp.onDidAccept(() => { /* qp.selectedItems */ });
qp.show();

// Webview panel (Tier 2 — declare in contributes.webviewPanels)
import { createWebviewHtml } from "@sindri/api/helpers";

const panel = sindri.ui.registerWebviewPanel(
  { id: "my-ext.panel", title: "My Panel", defaultDock: "right-bottom" },
  {
    getHtml(_ctx) {
      // createWebviewHtml generates the standard null-origin-safe HTML shell —
      // loads dist/webview.js and dist/webview.css via sindri-resource://
      return createWebviewHtml("my-ext.panel");
    },
    async onMessage(msg) {
      if ((msg as { type: string }).type === "ready") {
        panel.postMessage({ hello: "world" });
      }
    },
  },
);
```

#### `sindri.output`

```typescript
const channel = sindri.output.createOutputChannel("My Extension");
channel.appendLine("Extension activated.");  // appears in Extension Logs panel
```

`console.log` / `console.warn` / `console.error` are automatically captured to a `"console"` channel for the extension — no setup needed.

#### `sindri.events`

```typescript
sindri.events.on("workspace.fileChanged", (payload) => { /* ... */ });
sindri.events.emit("my-ext.somethingHappened", { data: 42 });
```

### Webview panels

Webview panels run in a null-origin `<iframe>`. They cannot make direct DOM calls into the host and cannot use ES module imports (`<script type="module">`). The build pipeline produces IIFE bundles specifically for this constraint.

**Loading assets via `sindri-resource://`:**

```text
sindri-resource://<extension-id>/<relative-path>
```

The scheme serves files directly from the extension's `dist/` folder (and anywhere else under its install directory). It is **read-only** and has no CORS headers.

**Webview ↔ extension messaging:**

```typescript
// Inside the webview (browser context):
window.__sindri_postMessage({ type: "ready" });

// Inside the extension (onMessage handler):
async onMessage(msg) {
  panel.postMessage({ result: "pong" });
}

// Receiving in the webview:
window.addEventListener("message", (e) => {
  const data = e.data; // whatever panel.postMessage() sent
});
```

### Icon theme with inheritance

The fastest way to ship a new icon theme is to extend `sindri-community-icons-base` rather than authoring 46+ SVGs from scratch. Your extension supplies a palette; the runtime merges it with the base at activation.

**What you provide — a `manifest.json` only:**

```json
{
  "$schema": "../../manifest.schema.json",
  "id": "yourname.my-icons",
  "name": "My Icons",
  "version": "1.0.0",
  "publisher": "yourname",
  "description": "My palette on the community icon set.",
  "categories": ["File Icon Theme"],
  "permissions": [],
  "engines": { "sindri": ">=0.1.0" },
  "extends": "sindri.community-icons-base",
  "variables": {
    "folder-base": "#5e81ac",
    "semantic-1":  "#81a1c1",
    "semantic-2":  "#a3be8c",
    "semantic-3":  "#d08770",
    "semantic-4":  "#bf616a"
  },
  "contributes": {
    "iconThemes": [
      { "id": "my-icons", "name": "My Icons", "kind": "color", "path": "icons.json" }
    ]
  }
}
```

**Variable reference:**

| Variable | Affects | Slot colour |
| --- | --- | --- |
| `folder-base` | All folder icons | Your theme's primary folder colour |
| `semantic-1` | Docker, config, SQL, SVG, Markdown | Blue-ish tool/infra colour |
| `semantic-2` | Shell, `.env`, JSON, YAML, plain text | Green-ish data/env colour |
| `semantic-3` | Git, TOML, images, XML | Orange-ish VCS/media colour |
| `semantic-4` | Lock files, package manifests | Red-ish danger/lock colour |

Language logo icons (TypeScript, Python, Rust, React, Vue, …) **always** use their canonical brand colors — `variables` has no effect on them.

**To override a specific icon**, add an `icons/` folder to your extension with an SVG at the same filename as the base asset (e.g. `icons/folder.svg`). Your file fully replaces the base asset; everything else is still inherited.

**To override icon mappings** (add a new `fileExtension`, rename a `folderName`, etc.), add a partial `icons.json` to your extension root. The runtime merges it with the base `icons.json` — your values win on any key conflict.

See [`sindri-dracula-icons`](community-theme-collection/sindri-dracula/sindri-dracula-icons/) for a worked example.

### UI Icon Theme

A UI Icon Theme provides the glyphs used in the activity bar and toolbar.

```json
{
  "$schema": "../manifest.schema.json",
  "id": "yourname.my-ui-icons",
  "name": "My UI Icons",
  "version": "1.0.0",
  "publisher": "yourname",
  "description": "Compact icons for the Sindri activity bar.",
  "categories": ["UI Icon Theme"],
  "permissions": [],
  "engines": { "sindri": ">=0.1.0" },
  "contributes": {
    "uiIconPacks": [
      { "id": "my-ui-icons", "name": "My UI Icons", "path": "ui-pack.json" }
    ]
  }
}
```

`ui-pack.json` is a `UiIconPackDef` — a flat JSON object mapping panel IDs to inline SVG strings:

```json
{
  "id": "my-ui-icons",
  "name": "My UI Icons",
  "icons": {
    "explorer":   "<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 24 24'>...</svg>",
    "search":     "<svg ...>...</svg>",
    "git":        "<svg ...>...</svg>",
    "extensions": "<svg ...>...</svg>"
  }
}
```

Panel IDs: `explorer`, `search`, `git`, `debug`, `extensions`, `terminal`, `problems`, `output`, `settings`, `run`, `test`, `snippets`, `bookmarks`, `database`, `containers`, `remote`.

SVGs should use `viewBox='0 0 24 24'` and `fill="currentColor"` (or `stroke="currentColor"`) so the theme accent colour is applied automatically.

### Extension Pack

An Extension Pack installs a curated list of extensions in one click.

```json
{
  "$schema": "../manifest.schema.json",
  "id": "yourname.my-pack",
  "name": "My Pack",
  "version": "1.0.0",
  "publisher": "yourname",
  "description": "Everything you need for Rust development.",
  "categories": ["Extension Pack"],
  "permissions": [],
  "engines": { "sindri": ">=0.1.0" },
  "contributes": {},
  "extensionPack": [
    "sindri.rust-language-support",
    "yourname.my-theme"
  ]
}
```

Each extension in `extensionPack` is installed individually and can be updated or removed independently.

## IconThemeDef format reference

```json
{
  "id": "my-icons-color",
  "name": "My Icons",
  "kind": "color",

  "fileNames":      { ".gitignore": "git", "package.json": "json" },
  "fileExtensions": { "ts": "ts", "tsx": "tsx", "rs": "rs", "md": "md" },
  "languageIds":    { "typescript": "ts", "rust": "rs" },
  "folderNames":    { "src": "folder-src", ".git": "folder-git" },
  "defaults":       { "file": "file", "folder": "folder", "folderOpen": "folder-open" },

  "icons": {
    "file": {
      "svg":     "<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 16 16'>...</svg>",
      "monoSvg": "<svg ...>...</svg>"
    },
    "ts": { "svg": "<svg ...>...</svg>" }
  }
}
```

Resolution order (first match wins): `fileNames` → `fileExtensions` → `languageIds` → `defaults`.

| Icon field | Description |
| --- | --- |
| `svg` | Full-colour inline SVG — used for `"color"` kind and as fallback for `"mono"` |
| `monoSvg` | Monochrome SVG using `currentColor` — preferred for `"mono"` kind |

SVGs should use `viewBox='0 0 24 24'` and be self-contained (no external references).

## index.json format

`index.json` uses typed buckets so the marketplace and resolver don't need to tree-walk:

```json
{
  "name": "...",
  "description": "...",
  "homepage": "...",
  "extensions": [
    { "id": "sindri.community-icons-base", "path": "community-theme-collection/sindri-community-icons-base" }
  ],
  "packs": [
    { "id": "sindri.aurora-theme-pack", "path": "aurora-theme-pack" }
  ],
  "collections": [
    { "id": "sindri.community-theme-collection", "path": "community-theme-collection" }
  ]
}
```

| Bucket | Contains |
| --- | --- |
| `extensions` | Standalone leaf extensions — **not** pack members |
| `packs` | Extensions with `extensionPack` whose members are leaf extensions |
| `collections` | Packs-of-packs (e.g. a theme collection whose members are themselves packs) |

Pack members and collection members are **not** listed separately — the resolver discovers them by following `extensionPack`. A remote extension that can't be resolved is serialised as `{ "id": "…", "remote": true }` and the UI warns rather than hard-failing.

`index.json` is **generated** — run `bun run scripts/build-index.ts` from the `sindri-extensions/` directory after adding or changing an extension. Do not edit it by hand.
